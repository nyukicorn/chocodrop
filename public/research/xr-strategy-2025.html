ChocoDrop XR リサーチブリーフ

Meta Quest 3 を主軸に XR（VR/MR/AR）機能を追加するための市場調査、方向性、機能要件のまとめ。

1. マーケット & デバイス動向

エコシステム

  • Meta Horizon OS が ASUS・Lenovo など外部 OEM へ拡張予定。Quest 4 は 2027 年以降との見方で、2025-26 年は現行ハード＋パートナー端末で需要を維持する方針。1
  • Apple visionOS 2.4 で Apple Intelligence、Spatial Gallery、Guest User 拡張など “共有・生成” 系機能が揃い、空間 UX 競争が可視化。2
  • Samsung Galaxy XR が $1,799 で発売、Snapdragon XR2+ Gen 2・109° FOV・即日出荷で「Vision Pro より軽く安い」ポジションを確立。3

ブラウザ & SDK

  • Quest Browser v39 で WebXR Shared Spaces（共通座標）が実験提供され、ローカル複数台での自動位置合わせが可能に。4
  • Niantic 8th Wall Runtime 2.x が Asset Lab、物理エンジン刷新、デスクトップアプリを追加し、WebXR→ネイティブ書き出しの導線が強化。5

半導体 & 未来性

  • Qualcomm XR2+ Gen 2 は 4.3K/eye・12 カメラ・15% GPU 向上で MR パススルー精度を改善。Samsung / Google 向け高性能 HMD の標準 SoC として提示。6
  • IDC は 2025 年の AR/VR+スマートグラス市場 14.3M 台（+39.2%）を予測し、MR/ER が VR 単体より高速に成長すると試算。7

2. インサイトと示唆

  • XR = 混在体験へのシフト：各社とも「現実とのブレンディング」「AIアシスタンス」軸で共通しており、ChocoDrop でも WebXR/Three.js ベース UI を “空間 UI” として再構築する必要あり。
  • WebXR の戦略的価値：Quest Browser の共有空間と 8th Wall/Lightship の VPS 機能が揃い、ブラウザ配信でもマルチユーザー MR が実現可能。アプリ審査を経ずに 192.168.1.15 で配信できるため、現行スタックを活かせる。
  • ブラウザ→ネイティブ輸出：Samsung / Apple / Meta で API 差異が広がるため、WebXR＋OpenXR 層を標準化しつつ、将来的に 8th Wall の Native Export や Capacitor を介したハイブリッド配信を検討。

3. 必要機能リスト（初期スコープ案）

  1. WebXR ブリッジ：Quest Browser で自動的に MR セッションへ入るためのエントリーページと、機器判定に応じたフォールバック。
  2. パススルー UI：Three.js レイヤーに MR 透過（`XRCompositionLayer` & `alpha: true`）を適用し、現実背景上で ChocoDrop HUD を配置。Apple/Samsung 向けには Safari/Samsung XR Browser の互換実装を抽象化。
  3. 共有アンカー：Quest Shared Spaces + WebRTC でプレゼンス同期。Apple visionOS では SharePlay/GroupActivities 併用を想定。
  4. Spatial Input：ハンドトラッキング操作（pinch/drag）とコントローラ操作両対応。XR2+ Gen 2 の 12 カメラ構成や Apple Hand Tracking API を意識し、入力抽象層を整理。
  5. AI アシスト接合：visionOS の Writing Tools や Meta Ray-Ban の音声アシストを踏まえ、ChocoDrop のシーン生成機能に音声→LLM→Three.js の最短パスを用意。
  6. 192.168.1.15 配信基盤：HTTPS + WebSocket/TCP 43110 の現行デーモンを XR ルームごとにプロキシし、Quest/visionOS から同一プライベートネットでアクセスできるようにする。

4. 実装へのステップ

短期（〜2週間）

  • 192.168.1.15 上で HTTPS 証明書と WebXR Feature Policy を整備。
  • 現行 Three.js UI を WebXR 対応（`navigator.xr.requestSession('immersive-vr')`）。
  • Quest Browser Shared Spaces 実験を有効化し、2 台での座標同期 PoC。

中期（〜6週間）

  • 8th Wall Runtime 2.x で Asset Lab / Native Export の試行と、社内テンプレート化。
  • visionOS Safari/SharePlay、Galaxy XR Browser（Android XR）の互換検証。
  • LLM ベースのシーン生成フローを音声入力込みで Quest / visionOS 両対応に。

長期（Q1 2026〜）

  • メッシュ共有（VPS/Lightship）と物理ベース編集の統合。
  • Qualcomm XR2+ Gen 2 対応 HMD（Samsung XR 等）での 4K/eye レンダリング最適化。
  • App Lab / Apple Vision Pro App Store へのハイブリッド配信を視野に入れた署名フロー整備。

5. デプロイ留意点（Meta Quest 3）

  • Meta Browser Flags: chrome://flags/#webxr-shared-spaces を有効化し、社内 Quest へ共通イメージを配布。
  • ホスト: 192.168.1.15 は https://192.168.1.15 で自己署名 → Quest 信頼済み証明書としてサイドロード。
  • アクセス制御: 他チームと競合しないよう、XR ポートレンジを 443 + /xr-demo に固定。

6. 次のアクション

  • 本ブリーフを基に、XR 機能の詳細タスクをプランモード（Codex Plan Tool）で確定する。
  • デモ用ブラウザ HTML（Quest 3 試験ページ）を public/xr-demo/index.html に配置し、社内テスト URL を共有。
  • 2 週間ごとに市場アップデートを反映し、Feature リストをローリングリビュー。
